# French Localization — Execution-Ready Spec

French-only localization of My-Pokedex. This is a **data** localization job (official French
Pokémon names differ: Bulbasaur → Bulbizarre), not just UI translation. Native-French review
of all copy; every UI string lives in locale files. **No English may ever leak onto a `/fr/`
page.**

This spec is written against the **actual** architecture (verified, not assumed):
- **Next.js 16.2.6, Pages Router, React 19, TypeScript.** No `output: 'export'`; standard
  `next build`.
- **Fully SSG**: every route uses `getStaticProps`/`getStaticPaths` with `fallback: false`;
  all 1025 detail pages are prebuilt. HTML is fully server-rendered → **the "client-side
  fetch kills SEO" risk does NOT apply**; just mirror the SSG pattern for `/fr/`.
- **Data is fetched live from PokéAPI at BUILD time** and baked into the static output.
  **There is no committed dataset file.** French names/types/genus/flavor already ride inside
  the species/type payloads the build already fetches — so FR adds ~no extra API calls for
  those (abilities may need a few).
- **EN detail slugs are numeric**: `/details/{id}`. **These URLs never change.**
- **No i18n today**: no locale files, no i18n lib, no `/fr/` route. `<Html lang="en">` in
  `pages/_document.tsx`; `SITE_LOCALE="en_US"` in `constants/Seo.ts`. **All UI copy is inline
  hardcoded English** — a string catalog must be introduced from scratch.
- **SEO plumbing**: meta/OG/canonical centralized in `ui/components/Header/Header.tsx`;
  JSON-LD in `utils/structuredData.ts`; sitemap hand-generated by
  `scripts/generateSitemap.mjs`; `public/robots.txt` allow-all. **hreflang does not exist yet.**
- Production origin `https://www.my-pokedex.com` (`constants/Seo.ts`, sitemap, robots).

## Key design decisions (locked)

1. **EN stays at root, URLs 100% unchanged.** French lives under `/fr/`.
2. **French detail URLs use name-slugs**: `/fr/pokemon/{slug}` where slug = accent-stripped,
   lowercased official French name (`bulbizarre`, `electhor`, `salameche`). SSG, `fallback:
   false`, build-time PokéAPI fetch — a 1:1 mirror of the EN build model.
3. **No numeric→slug redirect.** The EN→FR language switcher already has the `id` and links
   straight to the correct slug via a build-time slug↔id map, so nobody lands on a bare
   numeric FR URL. (Do not build the redirect layer.)
4. **Route approach: a separate FR route tree** (`pages/fr/pokemon/[slug].tsx`,
   `pages/fr/index.tsx`), NOT Next's built-in `i18n` config — because EN is numeric and FR is
   name-slug, the two schemes differ and built-in locale routing can't express that cleanly.
5. **Resolution order for every displayed French field**: PokéAPI `fr` → repo overrides file
   → **loud build/coverage failure**. Never silently fall back to English on `/fr/`.

## STEP 1 — French dataset (extend the build-time fetch)

Extend the existing formatters (`utils/pokemonFormatter/`) and fetch layer
(`services/fetchPokemons/`) to also extract French fields alongside English. Keep the
build-time fetch model; **cache aggressively** (respect PokéAPI fair-use — never per-request).

Pull French from the payloads already fetched wherever possible:
- **Names**: `pokemon-species.names[]` where `language.name === "fr"` (Bulbizarre, Salamèche…).
- **Genus**: `pokemon-species.genera[]` fr ("Pokémon Graine").
- **Flavor text**: `pokemon-species.flavor_text_entries[]` filtered to fr, picking the **most
  recent game version** that has a French entry (whitespace/`\f` normalized).
- **Types**: `/type.names[]` fr (Feu, Eau, Plante, Électrik…).
- **Abilities**: `names[]` fr for the name. For the description, `effect_entries[]` fr if
  present, else fall back to `flavor_text_entries[]` fr. (Coverage is spotty — that's what the
  overrides + coverage check exist for.)
- **Everything else displayed**: stats, egg groups, forms, version names, etc. — use each
  endpoint's `names[]` fr.

Add a **localized-name field** to the dataset shape (`IFullPokemon`/`IBasicPokemon`) so both
the FR pages and FR search can use it.

## STEP 2 — Slugs, routing & search

**Slugify util** (`utils/slugify.ts`): lowercase → `normalize('NFD')` strip diacritics →
replace non-alphanumerics with `-` → collapse/trim dashes. `Électhor → electhor`,
`M. Mime → m-mime`, `Flabébé → flabebe`, `Porygon-Z → porygon-z`.

**Collision guard**: build a slug↔id map at build time from the French names; **fail the
build loudly if any two names collide.** Across the base 1025 the only real collision is
`Nidoran♀`/`Nidoran♂` → both `nidoran`; disambiguate to `nidoran-f`/`nidoran-m`. The guard
future-proofs new generations.

**Routing**:
- `pages/fr/pokemon/[slug].tsx`: `getStaticPaths` emits every French slug; `getStaticProps`
  resolves slug→id via the map, then fetches/formats (French-aware).
- `pages/fr/index.tsx`: French home/grid mirror.
- Prev/Next nav uses id arithmetic, then maps id→slug for the FR href.

**Search on `/fr/`**: match French names **accent- and case-insensitively**
(`électhor` = `electhor`) by normalizing both query and candidate; **also match English names
as a fallback**. Extend `utils/pokemonTypes/filtering.ts` / `hooks/useFiltering.ts` to filter
on the localized name field.

**UI chrome** → FR locale files (see below): Types, Statistiques, Faiblesses, Évolutions,
Taille, Poids, filters, search placeholder, pane labels ("Pokedex entry" → "Description
Pokédex", "Base stats" → "Statistiques de base", "Type effectiveness", "Profile", etc.).

## STEP 3 — Hardcoded FR overrides + coverage check

**Overrides file** `locales/fr-overrides.json`, structured by entity type + id/slug, e.g.:
```json
{
  "abilities": { "169": { "description": "…" } },
  "flavorText": { "905": "…" },
  "genus": { "1017": "…" }
}
```

**In-scope displayed fields (bounded — the coverage check ONLY inspects these):**
Pokémon name · genus · one flavor text · each type name · each ability name · each ability
description · stat labels · egg-group names · form names · type-effectiveness/damage labels.
(Do NOT scan the entire raw PokéAPI graph — that's unbounded. Scope = what renders on a page.)

**Coverage command** `npm run check-fr-coverage` (script in `scripts/`):
- Fetches from PokéAPI (cached/rate-limited, fair-use) and walks the in-scope field set for
  all 1025 + referenced entities.
- For every field missing a French value from both PokéAPI **and** the overrides file, it
  **scaffolds a stub entry** in `locales/fr-overrides.json` containing the **English source
  text as a reference comment/value**, keyed by entity+id.
- Then **exits non-zero with a complete, grouped report** of every unfilled stub, and stays
  red until all stubs are filled. (Mickael fills the stubs by hand.)
- Wired into `prebuild` so a data refresh (new gen / new Pokémon) surfaces new gaps the same
  way. The FR build itself also hard-fails if a rendered field would resolve to English.

## STEP 4 — SEO / GEO structure

- **hreflang** (build from scratch, in `Header.tsx`): reciprocal per detail page and index —
  `en`, `fr`, `x-default → English`. EN↔FR pairing via the id↔slug map.
- **Titles/metas**: FR around *"Pokédex en français — stats, types, faiblesses, évolutions"*;
  per-Pokémon titles use the French name. Translate `og:title`/`og:description`.
- **`<html lang="fr">`** and **`og:locale = fr_FR`** on FR pages (parameterize `_document`/
  `Header`; keep EN as-is).
- **Self-referencing `rel=canonical`** on every `/fr/` page → its own French-slug URL. Never
  canonical to the English version.
- **Sitemap**: extend `scripts/generateSitemap.mjs` to emit `/fr/pokemon/{slug}` + `/fr/`,
  annotated with reciprocal `xhtml:link` hreflang alternates. Confirm `robots.txt` doesn't
  block `/fr/` and there's no stray `noindex`. Keep **GPTBot, ClaudeBot, PerplexityBot**
  allowed.
- **Rendering**: FR pages are SSG/server-rendered (already the model) — verify the HTML
  contains French text, not a client-fetch placeholder.
- **GEO opening line** on each FR detail page, including the English name:
  *"Bulbizarre (anglais : Bulbasaur) est un Pokémon de type Plante/Poison de la 1re
  génération…"* — captures French users who search the English name.
- **Localized image alt**: *"Sprite de Bulbizarre"*.
- **BreadcrumbList JSON-LD** with French names, `inLanguage: "fr"` (extend
  `utils/structuredData.ts`).
- **Locale-scoped internal linking** on `/fr/`: evolution chains, same-type lists, adjacent
  Pokédex numbers — all pointing to `/fr/` slugs.
- **Language switcher** linking to the SAME Pokémon in the other language (id↔slug map). **No
  IP/Accept-Language auto-redirects.**
- **AdSense snippets untouched**; FR pages inherit the existing consent/CMP gating (no
  ungated ads, no ad-free FR).

## STEP 5 — Verify

1. `npm run build` passes with `check-fr-coverage` **green** (zero unfilled stubs) — or hand
   Mickael the full stub list to translate.
2. Spot-check 5 Pokémon across generations for correct FR name / types / flavor text.
3. **Grep the rendered `/fr/` output** for known English strings ("Seed Pokémon", English
   flavor text, "Base stats") to prove nothing leaks.
4. Search `électhor` and `electhor` both resolve.
5. hreflang reciprocal on the index + one detail page; confirm the collision guard passed.
6. Confirm **no English URL changed**.
7. List the locale + overrides files for Mickael's review.
